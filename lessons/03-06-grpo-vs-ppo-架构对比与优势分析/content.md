# 3.6 GRPO vs PPO：架构对比与优势分析

## 课程概述

本课时对PPO（Proximal Policy Optimization）和GRPO（Group Relative Policy Optimization）进行全面的架构对比与优势分析。作为现代大语言模型RLHF训练的两大核心算法，它们各自代表了不同的设计哲学：PPO通过裁剪机制和价值网络实现稳定更新，而GRPO通过组内相对比较消除对价值网络的需求。在DeepSeek-R1等推理模型的训练中，GRPO展现出了显著的优势，但PPO在复杂多目标场景中依然不可替代。理解这两算法的架构差异、适用场景和各自优势，是选择合适RLHF技术路线的关键。

**学习目标**
- 掌握PPO与GRPO的完整架构组成，能绘制并解释两算法的网络结构图
- 理解两种算法的内存占用差异，以及这如何影响大模型训练的可扩展性
- 掌握GRPO组采样相比PPO off-policy校正的样本效率优势
- 对比分析两种算法的KL约束机制：PPO裁剪 vs GRPO参考策略KL惩罚
- 评估两种算法的训练稳定性，理解各自的优势来源
- 根据实际场景（资源约束、精度要求、任务特性）做出算法选择
- 了解混合策略的前沿研究方向

**前置知识**：PPO算法详解（裁剪机制、KL散度）、GRPO详解（组计算、优势估计）、策略梯度定理、Actor-Critic架构

## 1. 架构组成对比

### 1.1 网络结构对比

PPO和GRPO在架构上的最根本区别在于网络组件的数量和功能分配：

| 组件 | PPO | GRPO |
|------|-----|------|
| **策略网络** | Policy Model（训练） | Policy Model（训练） |
| **旧策略网络** | Old Policy（冻结，用于重要性采样） | 与Policy共享，无需独立存储 |
| **价值网络** | Critic/Value Model（训练） | 无需独立网络 |
| **奖励网络** | Reward Model（冻结） | Reward Model（冻结） |
| **参考网络** | 无独立参考网络 | Reference Model（冻结，KL约束） |
| **优势估计** | GAE（需value网络） | Group内相对比较（无需value网络） |
| **网络总数** | **4个** | **2个** |

```
┌─────────────────────────────────────────────────────────────────┐
│                        PPO 架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│              │                │                                 │
│              │                └──▶ Value Model ──▶ Value          │
│              │                         │                        │
│              ▼                         ▼                        │

│              │                                                 │

│   • 需要训练Value网络（额外参数）                        │

```

### 1.3 内存占用对比

对于大规模语言模型，内存占用是决定训练可行性的关键因素：

| 模型规模 | PPO内存占用（GB） | GRPO内存占用（GB） | 节省比例 |
|---------|-----------------|------------------|--------|
| 7B 参数 | ~56 | ~40 | ~29% |
| 13B 参数 | ~104 | ~72 | ~31% |
| 70B 参数 | ~560 | ~380 | ~32% |
| 405B 参数 | ~3200 | ~2100 | ~34% |

**计算说明**：
- PPO需要存储：Policy(FP16) + Old Policy + Value + Reference + Optimizer States + Activations
- GRPO需要存储：Policy + Reference + Optimizer States + Activations
- Value网络（参数量约为Policy的10-15%）被完全移除

## 2. 样本效率对比

### 2.1 PPO的Off-Policy数据处理

PPO虽然声称是on-policy算法，但实际上通过重要性采样修正实现了off-policy数据复用：

```
┌─────────────────────────────────────────────────────────┐
│           PPO Off-Policy 数据处理流程                   │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  1. 使用旧策略π_{θ_old}采样trajectory                   │
│                                                          │
│  2. 计算重要性采样比率：                                 │
│     r_t(θ) = π_θ(a_t|s_t) / π_{θ_old}(a_t|s_t)         │
│                                                          │
│  3. 通过裁剪限制比率范围：                               │
│     clip(r_t(θ), 1-ε, 1+ε)                             │
│                                                          │
│  问题：                                                  │
│  • 当策略更新幅度大时，r_t(θ)可能极度偏离1              │
│  • 裁剪虽然保护了稳定性，但牺牲了数据效率               │
│  • 多次epoch复用同一批数据会导致过拟合                   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 2.2 GRPO的组采样优势

GRPO通过组采样从根本上改变了样本利用方式：

```
┌─────────────────────────────────────────────────────────┐
│           GRPO 组采样机制                               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  对于每个query q：                                       │
│                                                          │
│  1. 使用π_{θ_old}生成G个不同输出{o₁, o₂, ..., oG}      │
│                                                          │
│  2. 所有G个输出共享相同的输入q：                         │
│     - 相同的问题上下文                                   │
│     - 不同的随机采样路径                                 │
│     - 保证样本间的可比性                                 │
│                                                          │
│  3. 组内相对优势计算：                                   │
│     Â_i = (r_i - mean(r)) / std(r)                      │
│                                                          │
│  4. 所有G个样本一次性用于策略更新                        │
│                                                          │
│  优势：                                                  │
│  • 零额外成本获得G倍数据密度                             │
│  • 组内共享上下文减少了估计方差                          │
│  • 相对比较天然解决了奖励尺度不一致问题                  │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 2.3 样本效率量化对比

| 指标 | PPO | GRPO |
|------|-----|------|
| 单次迭代样本数 | N（单输出） | G×N（G个输出） |
| 样本间上下文共享 | 无（各自独立采样） | 有（同query的G个输出） |
| 奖励基线估计 | 学习得到的Value网络 | 组内统计量 |
| 方差来源 | 策略采样 + Value估计误差 | 仅策略采样 |
| Off-policy修正需求 | 需要（重要性采样） | 不需要（同策略采样） |

## 3. KL约束机制对比

### 3.1 PPO的裁剪约束

PPO使用裁剪机制隐式实现KL约束，核心是限制策略比率的变化：

```
┌─────────────────────────────────────────────────────────┐
│           PPO 裁剪机制                                   │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  目标函数：                                              │
│  L^{CLIP}(θ) = E[min(r_t(θ) * A_t,                     │
│                      clip(r_t(θ), 1-ε, 1+ε) * A_t)]    │
│                                                          │
│  其中：r_t(θ) = π_θ(a_t|s_t) / π_{θ_old}(a_t|s_t)     │
│                                                          │
│  裁剪效果：                                              │
│  • 当 r_t(θ) > 1+ε，裁剪为 1+ε（限制策略增加幅度）      │
│  • 当 r_t(θ) < 1-ε，裁剪为 1-ε（限制策略减少幅度）      │
│                                                          │
│  隐式KL控制：                                           │
│  • 策略比率限制在[1-ε, 1+ε]间接限制了KL散度             │
│  • 但无法精确控制KL的绝对值                             │
│  • KL ≈ (r_t - 1)²/2（当r_t接近1时）                   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 3.2 GRPO的双层KL控制

GRPO采用更直接的KL约束机制，在奖励计算和目标函数两个层面实施：

| 层次 | 公式 | 作用 |
|------|------|------|
| **微观KL控制** | r_t = r_φ(q, o_{≤t}) - β log(π_θ/π_ref) | 每个时间步的奖励调整 |
| **宏观KL控制** | L^{GRPO} = L^{CLIP} - β·D_{KL}(π_θ‖π_ref) | 目标函数层面的整体约束 |

### 3.3 约束机制对比总结

| 特性 | PPO裁剪 | GRPO参考KL |
|------|---------|-----------|
| 约束形式 | 隐式（通过比率裁剪） | 显式（直接KL散度） |
| KL计算 | 无需显式计算 | 需要计算D_{KL}(π_θ‖π_ref) |
| 控制精度 | 间接，难以精确 | 直接，可精确设定目标值 |
| 超参数 | ε（裁剪范围） | β（KL惩罚系数）+ 自适应调整 |
| 适用场景 | 通用，默认选择 | 需要精确控制策略偏离的场景 |

## 4. 训练稳定性对比

### 4.1 PPO的稳定性来源

PPO的稳定性主要来自三个方面：

1. **裁剪机制**：限制每次更新的最大幅度，防止灾难性策略偏移
2. **价值网络**：提供相对稳定的基线估计，减少奖励方差
3. **多次epoch更新**：每批数据多次复用，充分挖掘样本价值

```
PPO训练稳定性因素：
├── 裁剪限制：每次更新幅度 ≤ ±20%（默认ε=0.2）
├── Value基线：优势估计方差减少约 50-70%
└── 信任域：隐式限制策略在合理范围内探索
```

### 4.2 GRPO的稳定性来源

GRPO通过不同的机制实现稳定性：

1. **组内标准化**：自适应基线消除不同问题难度的影响
2. **参考策略约束**：防止策略过度偏离初始能力分布
3. **无需Value网络**：避免了Value估计误差引入的不稳定因素

```
GRPO训练稳定性因素：
├── 组内基线：std(r)自动适应不同问题难度
├── 参考约束：D_{KL}(π_θ‖π_ref) < δ 硬约束
└── 无Value误差：消除额外的噪声来源
```

### 4.3 稳定性对比分析

| 场景 | PPO | GRPO | 分析 |
|------|-----|------|------|
| 简单奖励任务 | 稳定 | 稳定 | 两者差异不明显 |
| 长序列推理 | Value估计误差累积 | 组内标准化更鲁棒 | GRPO更优 |
| 多目标平衡 | 需要额外机制 | 天然支持 | GRPO更灵活 |
| 稀疏奖励 | 可能收敛慢 | 组比较提供更强信号 | GRPO更有效 |
| 大幅策略更新 | 裁剪限制 | KL约束限制 | 取决于具体配置 |

## 5. 适用场景与选择指南

### 5.1 GRPO的优势场景

**DeepSeek-R1推理模型训练案例**：

| 任务 | GRPO优势 | 原因分析 |
|------|---------|---------|
| 数学推理（MATH benchmark） | 显著优于PPO | 组内相对比较更适合验证最终答案 |
| 代码生成（Codeforces） | 明显优势 | 多解路径的相对评估比绝对评分更有效 |
| 思维链推理 | 突出优势 | 组内比较自然处理长序列的稀疏奖励 |

**DeepSeek-R1论文结果**：
```
MATH-500 benchmark:
• GRPO: 88.3% (DeepSeek-R1)
• PPO baseline: 81.4%
• 提升幅度: +6.9pp

Codeforces:
• GRPO: 76.3% pass@1
• PPO baseline: 68.7%
• 提升幅度: +7.6pp
```

### 5.2 PPO的优势场景

| 场景 | PPO优势 | 原因分析 |
|------|---------|---------|
| 多目标奖励平衡 | 更精确的奖励塑形 | 可通过Value网络学习复合奖励 |
| 复杂奖励函数 | 更稳定 | 裁剪提供额外的保护层 |
| 对抗性环境 | 更鲁棒 | 多次epoch复用提供更好的策略探索 |
| 安全性关键应用 | 更保守 | KL约束的显式控制更容易审计 |

### 5.3 选择决策树

```
算法选择决策树：

输入：你的训练场景
│
├─▶ 是否需要精确控制策略偏离程度？
│   ├─ 是 → PPO（显式KL控制）
│   └─ 否 → 继续判断
│
├─▶ 奖励是否是相对可比较的？
│   ├─ 是 → GRPO优先
│   └─ 否 → PPO可能更合适
│
├─▶ 是否有计算资源限制？
│   ├─ 是 → GRPO（更少参数，更低内存）
│   └─ 否 → 两者皆可
│
└─▶ 是否需要过程监督（每步评估）？
    ├─ 是 → PPO（GAE更适合过程监督）
    └─ 否 → GRPO（结果监督更高效）
```

## 6. 前沿：混合方法探索

### 6.1 现有研究趋势

近年来，研究者开始探索融合两者优势的混合方法：

| 方法 | 核心思想 | 结果 |
|------|---------|------|
| PPO+GRPO混合 | 用GRPO的优势估计替代PPO的GAE | 减少Value网络训练不稳定性 |
| 分层策略 | 粗粒度用PPO，细粒度用GRPO | 在复杂任务中效果提升 |
| 自适应切换 | 根据训练阶段选择算法 | 早期GRPO快速收敛，后期PPO精细调优 |

### 6.2 未来方向

1. **自适应算法选择**：根据任务特性自动选择合适的算法
2. **优势融合**：将PPO的裁剪稳定性与GRPO的样本效率结合
3. **分布式组采样**：在大规模集群上实现更高效的组采样
4. **多模态扩展**：将混合方法扩展到多模态RLHF场景

## 本章小结

1. **架构差异**：PPO需要4个网络（Policy、Old Policy、Value、Reward），GRPO仅需2个（Policy、Reference），这使GRPO在内存效率上具有显著优势

2. **样本效率**：GRPO通过组采样在单次迭代中产生G倍的有效样本，且组内共享上下文减少了方差；PPO需要通过重要性采样处理off-policy数据

3. **KL约束**：PPO使用裁剪机制隐式约束，GRPO使用参考策略的显式KL惩罚；后者提供了更精确的控制但需要额外的计算

4. **训练稳定性**：PPO的稳定性来自裁剪和Value网络，GRPO的稳定性来自组内标准化和参考约束；两者在不同场景下各有优势

5. **选择原则**：推理任务、相对可比较的奖励、资源受限场景优先选择GRPO；多目标平衡、复杂奖励函数、安全关键应用优先选择PPO

6. **前沿方向**：混合方法正在探索如何融合两者优势，未来可能出现更先进的统一框架
