# 15.1 垂直领域SFT：医疗/法律场景案例

## 课程概述

本课程深入探讨垂直领域监督式微调（Supervised Fine-Tuning, SFT）的实战应用，以医疗和法律两个高风险、高价值领域为案例。学生将学习如何在保护数据隐私的前提下，构建符合行业规范的专业领域微调数据集，设计有效的微调策略，并建立严格的安全评估体系。

## 学习目标

- 理解垂直领域SFT的核心挑战与价值
- 掌握医疗领域数据收集、处理与合规的最佳实践
- 掌握法律领域文档处理与实体识别技术
- 学会设计领域特定的评估基准与方法
- 了解垂直领域模型部署的安全 guardrails

## 前置知识

- 了解SFT的基本原理（参考课程14.1）
- 熟悉Python编程与数据处理
- 了解大语言模型的基本架构

## 1. 为什么需要垂直领域微调

### 1.1 通用模型的局限性

通用大语言模型虽然在广泛任务上表现出色，但在垂直领域面临三个核心挑战：

**领域知识缺失**：通用模型的知识来自互联网规模的文本预训练，对医疗、法律、金融等专业化领域的深度知识掌握有限。以医疗为例，通用模型可能知道"糖尿病"的基本概念，但缺乏对最新临床指南、药物相互作用、诊疗路径等专业知识。

**术语理解偏差**：专业领域有大量专用术语和缩写，这些术语在通用语料中出现的频率低，导致模型理解不准确。例如医疗领域的"ECMO"（体外膜肺氧合）、法律领域的"善意取得"，都需要专门的训练才能正确理解。

**表达规范差异**：医疗文书、法律文本有其独特的格式和表达规范，通用模型生成的内容往往不符合专业写作标准。

### 1.2 高风险场景的特殊要求

医疗和法律是典型的高风险场景，输出质量直接关系到人身安全和法律权益：

**容错率极低**：医疗场景中，一个错误的诊断建议可能导致患者延误治疗；法律场景中，一句错误的法律意见可能让当事人承担巨大经济损失。与娱乐聊天不同，这些领域不允许"差不多就行"。

**可追溯性要求**：专业领域需要模型输出能够追溯到权威来源，引用具体的法规条文或临床指南。这要求模型具备强大的引用和推理能力。

**隐私保护严格**：医疗数据受HIPAA等法规保护，法律数据涉及商业秘密和隐私，都不能直接用于训练。

### 1.3 垂直领域微调的价值

通过垂直领域微调，可以实现：

- 准确使用专业术语和表达规范
- 理解领域特有的推理模式和决策流程
- 生成符合行业标准的输出格式
- 大幅降低 hallucination（幻觉）的风险
- 在专业基准测试上达到甚至超越人类专家水平

## 2. 医疗领域案例

### 2.1 数据收集与来源

医疗领域SFT的数据来源主要有四类：

**医学文献**：包括PubMed论文、临床指南、药品说明书、医学教科书等。这些文献通常以PDF或XML格式存储，需要专门的解析工具提取文本内容。

**临床笔记**：来自电子健康记录（EHR）系统的脱敏数据。由于涉及患者隐私，需要进行严格的匿名化处理，移除所有PHI（Protected Health Information）信息。

**医学问答**：如MedQA、Medical PeerQA等公开数据集，以及医疗机构内部积累的Q&A数据。问答数据对训练模型的问题理解和回答能力特别有效。

**医患对话**：经过脱敏处理的真实医患沟通记录。这类数据能帮助模型学习如何用通俗语言解释专业医学概念。

### 2.2 HIPAA合规与数据处理

处理医疗数据必须严格遵守HIPAA（健康保险便携与责任法案）规定：

**PHI识别与移除**：需要识别的PHI包括患者姓名、电话、邮箱、地址、社会安全号、医疗保险号、病历号、设备标识符、URL、IP等18种标识符。使用正则表达式和命名实体识别相结合的方式进行检测。

**去标识化标准**：满足HIPAA Safe Harbor标准的去标识化要求：移除所有直接标识符，并确保剩余信息无法重新识别个人身份。

**数据访问控制**：训练环境应与外部网络隔离，数据存储使用加密存储介质，访问记录需要完整审计。

**最小必要原则**：只使用完成任务所必需的最少数据，避免过度收集。

```python
# PHI识别示例
PHI_PATTERNS = {
    'name': r'\b[A-Z][a-z]+ [A-Z][a-z]+\b',
    'phone': r'\b\d{3}[-.\s]?\d{3}[-.\s]?\d{4}\b',
    'email': r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b',
    'ssn': r'\b\d{3}-\d{2}-\d{4}\b',
    'mrn': r'\bMRN[:\s]*\d+\b',
    'date': r'\b\d{1,2}/\d{1,2}/\d{4}\b',
}
```

### 2.3 医疗领域微调策略

**LoRA配置建议**：

医疗领域建议使用较低的rank（4-8）和较高的alpha（16-32），这样可以更好地保持原有模型的能力同时学习领域知识。目标模块建议包含：
- `q_proj`, `v_proj`：注意力机制的查询和值投影
- `k_proj`, `o_proj`：键和输出投影
- `gate_proj`, `up_proj`, `down_proj`：FFN层

**数据配比策略**：
- 40% 医学文献QA对：建立知识基础
- 30% 临床案例分析：培养诊断推理能力
- 20% 医患对话：学习沟通技巧
- 10% 药品/治疗指南问答：强化专业信息准确性

**温度参数**：医疗问答建议使用较低的temperature（0.3-0.5），减少幻觉风险，保证输出的确定性。

### 2.4 医疗领域评估基准

**医学考试基准**：
- USMLE（美国医师执照考试）：包含Step 1、Step 2、Step 3
- MedQA：来自中国医师执照考试的数据集
- PubMedQA：基于PubMed文献的问答

**评估指标**：
- 准确率（Accuracy）：回答正确的比例
- F1分数：precision和recall的调和平均
- 引用准确率：回答中引用的文献是否正确
- 一致性：多次回答的稳定性

**人工评估维度**：
- 临床推理的逻辑完整性
- 诊断建议的适当性
- 术语使用的准确性
- 患者沟通的清晰度

### 2.5 医疗AI的安全考量

医疗领域的 hallucination 是生死攸关的问题，需要多重安全机制：

**输出限制**：模型被严格限制不能给出最终诊断，只能提供建议并强调"请咨询专业医生"。

**不确定性表达**：当模型对答案不够确信时，应明确表达不确定性，建议进一步检查。

**专业边界**：模型被训练识别超出其能力范围的复杂病例，及时转诊建议。

**实时知识更新**：医疗知识不断更新，模型需要定期用最新文献进行增量训练。

## 3. 法律领域案例

### 3.1 法律数据来源与处理

**法院文书**：判决书、裁定书、调解书等。这类文档结构化程度较高，通常包含案号、法院名称、当事人信息、案情描述、判决理由和判决结果。

**法律合同**：包括各类商业合同、劳动合同、知识产权协议等。合同文本通常较长，需要关注条款的精确含义。

**案例法**：判例汇编、法律解释等。英美法系的判例法传统使其在法律推理中具有重要地位。

**法规条文**：各类法律法规的原文。中文法律体系主要依赖成文法典。

### 3.2 个人信息标记与移除

法律文档中的个人信息处理同样重要：

**当事人信息**：原告、被告、第三人的姓名（尤其需要注意未成年人姓名）、地址、联系方式。

**案件特定信息**：案号、法官姓名、律师姓名、证人信息等。

**商业敏感信息**：涉及商业秘密的内容需要特别处理。

```python
LEGAL_PHI_PATTERNS = {
    'case_number': r'\b\d{4}[年沪深]?\d{2,6}[刑民行商执]?第?\d+\b',
    'lawyer_name': r'(?:代理|委托)律师[：:]\s*([^\s]+)',
    'judge_title': r'审判长|审判员|陪审员|书记员',
    'company_name': r'(?:原告|被告|第三人)\s*[:：]\s*([^\s公司厂站社])',
}
```

### 3.3 法律领域微调特点

**法律推理能力**：法律领域特别强调推理过程，需要模型能够：
- 识别法律问题的类型（民事/刑事/行政）
- 引用适用的法律条文
- 分析案件事实与法律要件的对应关系
- 得出逻辑严谨的法律结论

**引用准确性**：模型需要准确引用法律条文，不能捏造法条编号。训练数据中的引用应经过严格校验。

**辖区差异处理**：中国法律体系与美国等英美法系存在本质差异，需要注意：
- 成文法 vs 判例法的推理模式差异
- 实体法与程序法的区分
- 不同法域的特殊规定

### 3.4 法律领域评估基准

**律师资格考试**：
- 中国：国家统一法律职业资格考试（司考）
- 美国：Bar Exam（各州不同）
- 题目类型包括选择题、案例分析、法律文书写作

**评估指标**：
- 实体法准确率：法律条文引用的正确性
- 程序法准确率：诉讼程序的合规性
- 推理逻辑性：法律推理步骤的完整性
- 文书规范性：法律文书的格式和用语标准

### 3.5 法律AI的特殊挑战

**法律不确定性**：很多法律问题在学术界和实务界都没有定论，模型需要学会表达这种不确定性。

**时效性问题**：法律法规不断更新，模型需要具备识别法条有效期的能力。

**价值冲突**：某些案件涉及多方利益的平衡，模型需要能够分析不同立场的合理性。

## 4. 跨领域通用模式

### 4.1 高质量Instruction数据构建

垂直领域SFT的核心是构建高质量的instruction-response对：

**Instruction设计原则**：
- 明确指定角色和任务背景
- 提供足够的上下文信息
- 使用专业的指令语言
- 包含明确的输出格式要求

**Response构建原则**：
- 内容准确，基于权威来源
- 逻辑清晰，推理步骤完整
- 表达专业，符合行业规范
- 引用明确，方便核实

### 4.2 领域词汇处理

专业领域通常有大量专用术语，需要特别处理：

**术语标准化**：建立领域术语词典，统一同义词的表述。

**缩写展开**：确保模型理解专业缩写的完整含义。

**跨语言对应**：医学和法律都有大量拉丁语/英语术语，需要建立对照表。

### 4.3 RAG增强考虑

在垂直领域，检索增强生成（RAG）常常与SFT结合使用：

**知识库构建**：将领域权威文档（指南、法规、判例）向量化存储。

**检索增强训练**：在训练时让模型学会利用外部知识库。

**实时检索部署**：推理时结合检索结果减少幻觉。

### 4.4 人类审查循环

垂直领域必须建立严格的人类审查机制：

**专家审核**：由领域专家审核训练数据和模型输出。

**红队测试**：专门测试模型在边界情况的表现。

**持续监控**：部署后持续监控输出质量，及时发现偏差。

## 5. 端到端工作流程

### 5.1 领域数据收集与清洗

```
数据源 → 格式转换 → PHI移除 → 质量过滤 → 结构化处理
```

**数据源整合**：来自多个来源的数据需要统一格式，建立标准化处理流程。

**质量评分**：使用规则和模型对数据进行质量评分，过滤低质量样本。

**多样性保证**：确保数据覆盖领域内的各类场景和边界情况。

### 5.2 Instruction数据集构建

```
原始文本 → 意图标注 → Instruction生成 → Response编写 → 质量审核
```

**自动生成**：使用大模型辅助生成Instruction，但需要专家审核。

**人工标注**：关键Response由领域专家撰写或审核。

**难度分级**：按照简单问答→案例分析→综合推理递进。

### 5.3 LoRA/QLoRA微调

```
基础模型选择 → LoRA配置 → 训练监控 → 早停策略 → 模型合并
```

**基础模型选择**：根据任务复杂度和硬件条件选择合适的模型规模。

**训练监控**：关注loss曲线、梯度范数、生成质量等指标。

**早停策略**：设置验证集，当指标不再提升时提前停止训练。

### 5.4 领域特定评估

```
基准测试 → 专家评估 → 红队测试 → 偏差检测 → 迭代优化
```

**自动化基准**：使用医学考试题、法律资格考试题等标准化测试。

**专家评估**：邀请领域专家对模型输出进行人工评分。

**安全边界测试**：专门测试可能导致严重后果的边界情况。

### 5.5 部署Guardrails

```
输入审核 → 模型推理 → 输出检验 → 风险提示 → 人工兜底
```

**输入过滤**：检测并拒绝可能导致危险输出的输入。

**输出检验**：对模型输出进行事实核查，标记不确定内容。

**安全提示**：在涉及生命财产安全的输出中添加必要的免责声明。

**人工接管**：在高风险场景保留人工决策通道。

## 总结

本课程通过医疗和法律两个典型垂直领域，深入分析了领域定制SFT的完整流程：

1. **数据层面**：理解领域数据的特殊性，建立严格的隐私保护和质量管理流程。

2. **技术层面**：掌握领域特定的微调策略，包括LoRA配置、数据配比、训练策略等。

3. **评估层面**：建立领域专属的评估基准，结合自动化测试和专家审核。

4. **安全层面**：认识到高风险领域对模型输出的严格要求，建立完善的guardrails机制。

垂直领域微调是让大语言模型真正产生实际价值的必由之路。通过本课程的学习，你应该能够设计和实现完整的垂直领域SFT方案。
