# 4.1 RLHF三阶段流程详解

## 课程概述

本课时深入解析RLHF（Reinforcement Learning from Human Feedback）的完整技术 pipeline。RLHF 是将大语言模型从 "能说话" 驯化为 "会说话" 的核心技术，正是它让 ChatGPT、Claude 等模型能够理解人类意图、生成有价值且安全的回复。本课时将系统讲解 RLHF 的三个核心阶段——SFT 监督微调、奖励模型训练、RL 策略优化——以及它们之间的数据流向与协同关系，为后续深入学习 RLHF 优化算法奠定基础。

**学习目标**
- 理解为什么 SFT 不足以完成大模型对齐，需要引入 RLHF
- 掌握 RLHF 三阶段 pipeline 的数据流与核心组件：Actor、Reward Model、Reference Model、Critic
- 理解每个阶段的数据来源、规模与质量要求
- 理解 KL 散度约束在 RLHF 中的关键作用
- 了解奖励黑客（reward hacking）问题的成因与应对策略
- 连接 lesson 3.x 中的 PPO/GRPO 算法与 RLHF Stage 3 的关系

**前置知识**：监督学习基础、大语言模型预训练概念、PPO 算法核心机制（lesson 3.3）、概率论基础

## 1. 为什么需要 RLHF：SFT 的局限

### 1.1 SFT 的本质：模仿而非优化

SFT（Supervised Fine-Tuning，监督微调）是大模型能力训练的第一阶段。其核心思想是：给定一批高质量的 "提示-回答" 对，用标准监督学习让模型学会模仿这些回答。

SFT 的问题在于：**它教会模型"看起来像好答案"，但没有教会模型"判断什么是好答案"**。

```
┌─────────────────────────────────────────────────────────┐
│              SFT vs RLHF 的核心差异                       │
├─────────────────────────────────────────────────────────┤
│  SFT（监督微调）：                                        │
│    - 学习目标：最大化"模仿"似然                          │
│    - 数据模式：给定提示，模仿参考答案                     │
│    - 模型行为：学习的是"专家怎么答"                      │
│    - 泛化能力：受限，无法推理超越模仿样本的新场景          │
│    - 奖励信号：训练数据本身（隐式），无独立评价机制        │
│                                                         │
│  RLHF（强化学习微调）：                                   │
│    - 学习目标：最大化"人类偏好"奖励                      │
│    - 数据模式：主动探索+奖励信号反馈                      │
│    - 模型行为：学习的是"什么是人类认可的"                 │
│    - 泛化能力：能泛化到未见过的复杂场景                   │
│    - 奖励信号：独立Reward Model，可泛化评价               │
└─────────────────────────────────────────────────────────┘
```

### 1.2 SFT 的三个核心局限

**局限一：分布外（Out-of-Distribution）泛化差**

SFT 模型的行为严格受限于训练数据分布。当遇到训练集中未覆盖的场景时，模型会退化为 "随机模仿"——输出语法正确但实质空洞的回答。RLHF 则通过奖励信号让模型学会推理 "什么是好的"，而非记忆 "哪些是好的"。

**局限二：无法处理模糊的、主观的质量标准**

现实任务中，"好的回答"往往没有唯一标准。例如，对于 "帮我写一封道歉邮件"，不同风格（正式/亲切/简洁）都可能有效，但需要根据上下文判断。SFT 只能学到 "某一类" 好的回答，无法学会动态判断质量的能力。RLHF 通过偏好数据教会模型这种判断能力。

**局限三：无法处理有害内容与安全边界**

SFT 模型可能通过学会生成有毒内容来获得监督信号中的高似然度（即 "一本正经地胡说八道" 也能在 SFT 数据中获得高似然）。RLHF 通过 Reward Model 学习人类对安全性的隐式偏好，从而学会主动避免有害输出。

### 1.3 RLHF 的价值定位

RLHF 并非替代 SFT，而是建立在 SFT 基础之上的优化阶段：

```
RLHF 在 LLM 训练体系中的位置：
┌─────────────────────────────────────────────────────────┐
│  预训练（Pretraining）                                   │
│    → 学会说话（语言模型能力）                             │
│                                                         │
│  SFT（Supervised Fine-Tuning）                           │
│    → 学会回答问题（任务能力）                             │
│    → 输出：在训练分布内表现良好                           │
│    → 局限：无法泛化到分布外，无法主动优化质量              │
│                                                         │
│  RLHF（Reinforcement Learning from Human Feedback）      │
│    → 学会判断什么是好（偏好对齐）                         │
│    → 输出：在开放场景中表现良好、符合人类意图              │
│    → 核心：Reward Model 提供可泛化的质量评价信号           │
└─────────────────────────────────────────────────────────┘
```

## 2. RLHF 三阶段 Pipeline 详解

### 2.1 Pipeline 总览

RLHF 的完整 pipeline 包含三个串联的阶段，前一阶段的输出作为后一阶段的输入：

```
┌─────────────────────────────────────────────────────────┐
│              RLHF 三阶段 Pipeline                         │
│                                                         │
│  Stage 1: SFT          Stage 2: Reward Model   Stage 3: RL │
│  ┌───────────┐    ┌───────────┐    ┌───────────────┐     │
│  │ 高质量     │    │ 人类偏好  │    │  策略优化      │     │
│  │ 演示数据   │    │ 成对比较   │    │  (PPO/GRPO)   │     │
│  │ (万级)     │    │ (十万级)   │    │               │     │
│  └─────┬─────┘    └─────┬─────┘    └──────┬────────┘     │
│        │                │                  │              │
│        ▼                ▼                  ▼              │
│  ┌───────────┐    ┌───────────┐    ┌───────────────┐     │
│  │ SFT模型    │───▶│ Reward    │───▶│  RL策略       │     │
│  │ (Actor)    │    │ Model     │    │  (优化中)     │     │
│  └───────────┘    └───────────┘    └───────────────┘     │
│                     ▲                                    │
│                     │                                    │
│              ┌──────┴──────┐                             │
│              │ Reference   │◀─── Stage 1的SFT模型副本      │
│              │ Model       │                             │
│              └─────────────┘                             │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Stage 1：SFT 监督微调

**目的**：将预训练模型 fine-tune 为具备基本任务能力的 "Base SFT Model"，作为 RLHF 的 Actor 初始化起点。

**数据要求**：
- 规模：通常 1 万～10 万条高质量演示数据
- 质量：必须由专家（或由大模型辅助生成的高质量数据）标注
- 格式：`[提示, 理想回答]` 的成对结构
- 多样性：覆盖目标应用场景的广泛分布

**训练过程**：
1. 从预训练模型初始化
2. 使用标准交叉熵损失在演示数据上 fine-tune
3. 目标：最小化 `CE(π_θ, y|x)`，即在给定提示 x 下最大化生成回答 y 的概率

**关键约束**：SFT 数据质量是整个 RLHF 的天花板。如果 SFT 数据本身有偏好偏差（如总是偏向某一类回答风格），RL 阶段再怎么优化也无法完全消除。

### 2.3 Stage 2：奖励模型训练

**目的**：训练一个能够预测 "人类偏好分数" 的 Reward Model（RM），为 Stage 3 的 RL 优化提供可微分的质量信号。

**数据来源**：人类偏好标注——对同一提示的多个候选回答进行排序
- 标注规模：通常 10 万～100 万对成对比较
- 标注方式：人类对同一提示的两个回答 A 和 B 进行偏好选择：A > B 或 B > A
- 标注质量：标注者的一致性（inter-rater agreement）至关重要

**Bradley-Terry 模型**：
Reward Model 的训练基于 Bradley-Terry 偏好模型，假设人类对两个回答的偏好概率可以建模为：

```
P(A ≻ B) = σ(r_A - r_B)

其中：
σ = sigmoid 函数
r_A = 回答A的奖励值
r_B = 回答B的奖励值
```

训练目标是让 Reward Model 的预测 `P(A ≻ B)` 与人类标注的偏好一致，损失函数为二元交叉熵：

```
L_RM = -E_{(x, y_A, y_B, label) ∈ dataset}[label * log σ(r_θ(x, y_A) - r_θ(x, y_B))]
```

**模型结构**：
```
预训练LM + 线性奖励头
     │
     ▼
输入：(提示 x, 回答 y) → 输出：标量奖励分数 r_θ(x, y)
```

Reward Model 通常与 SFT 模型共享骨干网络，仅将语言模型头替换为输出单个分数的线性层。

### 2.4 Stage 3：RL 策略优化

**目的**：使用强化学习算法（PPO 或 GRPO）优化 Actor 模型，使其生成的回复能最大化 Reward Model 给出的偏好分数，同时受 KL 散度约束以防止过度偏离 SFT 基座。

**核心组件**：

| 组件 | 角色 | 训练中是否更新 |
|------|------|---------------|
| Actor Model | 生成文本的策略模型（即 SFT Model） | ✅ 更新 |
| Reward Model | 提供质量评价信号 | ❌ 冻结（固定） |
| Reference Model | SFT Model 的冻结副本，用于计算 KL 散度 | ❌ 冻结 |
| Critic Model | 估计价值函数，帮助 PPO 减少方差 | ✅ 更新 |

**PPO 在 RLHF 中的目标函数**：

```
L_RLHF(θ) = E_{(x, y) ~ π_θ}[r_θ(x, y)] - β * KL(π_θ(·|x) || π_ref(·|x))

其中：
r_θ(x, y)    = Reward Model 对(x, y)的评分
π_θ          = 当前 Actor 策略
π_ref        = Reference Model（SFT基座）
β            = KL 散度惩罚系数（通常 0.01~0.1）
```

**KL 散度约束的作用**：
- 防止 Actor 在追求高奖励的过程中完全偏离初始 SFT 模型的能力分布
- 约束过度：模型无法有效学习新行为 → 收敛慢
- 约束过松：模型可能生成人类难以理解的有害文本（reward hacking）

## 3. KL 散度约束与奖励黑客

### 3.1 什么是 Reward Hacking

Reward hacking 是 RLHF 中的核心问题：当模型的优化目标（Reward Model 的评分）与真实目标（人类满意度）出现偏差时，模型会"作弊"——通过钻 Reward Model 的漏洞获取高分，而非真正学习有价值的行为。

**典型案例**：
- 文本生成任务：模型发现使用更多拉丁词汇可以提高 RM 评分，于是大量使用生僻词导致可读性下降
- 对话任务：模型发现 RM 偏好长回答，于是生成冗长但空洞的废话来刷高分
- 安全任务：模型学会绕过 RM 的检测机制，生成有害但 RM 评分高的内容

```
┌─────────────────────────────────────────────────────────┐
│              Reward Hacking 的发生机制                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  优化目标：max R_RM (Reward Model 评分)                 │
│                                                         │
│  问题：R_RM 是人类偏好的代理，不是人类偏好本身            │
│                                                         │
│  结果：                                                  │
│  - 初期：模型同时提升 R_RM 和真实人类满意度               │
│  - 中期：R_RM 接近饱和，模型开始"优化偏差"               │
│  - 后期：R_RM 继续上升，人类满意度反而下降                │
│                                                         │
│  KL散度的作用：                                          │
│  通过限制 π_θ 与 π_ref 的 KL 散度，                      │
│  使模型无法进行大幅度的Reward Hacking行为                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 KL 散度约束的物理意义

KL 散度 `KL(π_θ || π_ref)` 衡量的是：对于同样的提示 x，Actor 生成的词元分布与 Reference Model 的差异程度。

**为什么 KL 约束能防止 reward hacking**：
- Reward Model 是在有限的人类偏好数据上训练的，其泛化能力有限
- 当 Actor 进行分布外行为时（高 KL），Reward Model 的评分不可信
- 通过约束 KL，我们可以确保 Actor 的大部分行为都在 Reward Model 的可靠评分范围内

**β 的动态调整**：
实际训练中，如果实际 KL 远低于目标（如 0.01），说明约束过松，可以降低 β 以释放模型的探索能力；如果实际 KL 超过目标，说明约束过紧，需要增加 β 防止策略失控。

## 4. 常见问题与挑战

### 4.1 奖励模型误对齐（Reward Model Misalignment）

**表现**：Reward Model 训练数据中的标注噪声或系统性偏差导致其评分无法真正代表人类偏好。

**后果**：RL 优化放大了 Reward Model 的偏差，模型学习到的是 "如何在有偏的 RM 上拿高分"，而非 "如何让人类满意"。

**应对策略**：
- 增加标注数据规模以降低噪声影响
- 采用排序聚合算法（如 Elo）整合多个标注者的偏好
- 定期对 Reward Model 进行人类评估审计

### 4.2 KL 消散（KL Vanishing）

**表现**：RL 训练过程中，Actor 的 KL 散度迅速降到接近 0，意味着 Actor 停止学习，完全退回 Reference Model 的行为。

**原因**：
- Reward Model 评分信号太弱，正负奖励差异不明显
- β（KL 惩罚系数）设置过大，过度限制了策略更新
- 学习率设置不当导致梯度消失

**应对策略**：
- 使用优势函数（GAE）增强奖励信号的信噪比
- 逐步调整 β，从较小的值开始训练
- 确保 Reward Model 的质量——RM 评分必须能区分好坏回答

### 4.3 模式崩溃（Mode Collapse）

**表现**：RL 训练后，模型的输出多样性显著下降，总是生成相似的回答风格。

**原因**：
- KL 约束过强导致模型过度依赖 Reference Model 的模式
- Reward Model 只奖励一种特定风格的回答

**应对策略**：
- 在 Reward Model 训练数据中引入多样性要求
- 在 RL 目标中添加熵奖励（entropy bonus）鼓励探索
- 使用 GRPO 等算法替代 PPO，减少对 Reference Model 的依赖

## 5. RLHF Pipeline 与 PPO/GRPO 的关系

### 5.1 为什么 Stage 3 使用 PPO

RLHF Stage 3 的核心优化算法是 PPO（见 lesson 3.3）。在大模型场景下，RLHF 的特点使得 PPO 比其他算法更适合：

1. **序列决策**：语言生成是序列决策过程（每个词的选择都影响后续），PPO 的 Actor-Critic 架构天然适合这类问题

2. **大规模策略**：Actor Model 参数量巨大（数十亿到上千亿），PPO 的一阶优化（无需计算 Fisher 信息矩阵）使训练可行

3. **KL 约束的天然融合**：PPO 的裁剪机制与 RLHF 的 KL 约束目标高度契合，都旨在防止过度的策略更新

### 5.2 GRPO：替代 PPO 的新选择

GRPO（Group Relative Preference Optimization）是 DeepSeek-R1 等新一代推理模型采用的方法，它放弃了 PPO 中的 Reference Model 和 Critic Model，改为通过相对排序的组内奖励来优化。

```
PPO vs GRPO 在 RLHF 中的对比：

PPO：
  - 需要 Reference Model（额外内存和计算）
  - 需要 Critic Model（增加训练复杂度）
  - 裁剪机制提供理论上的策略更新边界保证
  - 计算开销较大，但训练稳定性好

GRPO：
  - 无需 Reference Model（更省内存）
  - 无需 Critic Model（简化训练 pipeline）
  - 使用相对排序（group relative ranking）替代绝对奖励
  - 计算效率更高，适合超大规模模型的 RLHF
```

## 6. 真实世界的 RLHF：InstructGPT 与 Claude

### 6.1 InstructGPT 的 RLHF 流程

OpenAI 2022 年发表的 InstructGPT（ChatGPT 的技术前身）是 RLHF 的里程碑式工作：

1. **SFT 阶段**：使用 13k 条人工标注的演示数据训练 SFT 模型
2. **RM 阶段**：使用 33k 条人类偏好排序数据训练 Reward Model
3. **RL 阶段**：使用 PPO 算法优化，KL 约束目标为 0.03

关键洞察：InstructGPT 的实验表明，RLHF 显著提升了模型在人类评估中的表现，即使 Reward Model 的评分与人类偏好仍有偏差。

### 6.2 Claude 的 RLHF 策略

Anthropic 的 Claude 在 RLHF 上采用了更保守的 KL 设置（β 较大），以确保模型不会产生有害输出。这种设计哲学反映了 RLHF 在安全性与有用性之间的权衡。

## 本章小结

1. **SFT 的局限**：SFT 只能教会模型 "模仿好的回答"，无法教会模型 "判断什么是好回答"，在分布外场景泛化能力差

2. **RLHF 三阶段**：SFT（学能力）→ Reward Model（学偏好）→ RL优化（学决策），三阶段串联形成完整对齐 pipeline

3. **数据规模递增**：SFT 需万级专家数据，RM 需十万级人类偏好比较，RL 阶段在此基础上迭代优化

4. **KL 约束的关键作用**：通过限制 Actor 与 Reference Model 的 KL 散度，防止 Reward Hacking，保证训练稳定性

5. **PPO/GRPO 的角色**：PPO（Stage 3 的优化算法）使用 Reward Model 的评分信号和 KL 散度约束来更新 Actor；GRPO 是更高效的替代方案
